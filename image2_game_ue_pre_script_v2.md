# Image 2.0 对游戏交互设计师的价值

## 30 分钟 Pre 讲稿 2.0｜纯口播版

> UE 不是缺少思考，而是很多思考还没被看见。Image 2.0 的价值，是把脑子里的方向提前变成团队能讨论的画面。

## 一、开场：我们说的是同一个「爽」吗？｜00:00-05:00

> 排练锚点：00:00 开场；02:00 讲到“形容词共振”；04:00 收到“脑内截图”和主线。

大家好，今天我们聊一个在需求会上出现频率很高，但杀伤力也很强的词：爽。

这个词真的很神奇。策划说：“这个地方，再爽一点。”我们点头；UI 点头；动效点头；特效也点头。那一刻会议室非常祥和，仿佛项目已经离成功只差一个版本号。

但真正危险的地方，往往就在这里。

因为大家都说“懂了”的时候，脑子里可能根本不是同一张图。很多时候，大家只是对同一个形容词表达了礼貌性认可。

比如“更爽一点”。有人脑子里是更大的特效、更强的视觉爆点；有人想的是奖励反馈要更明显，最好一看就知道“我赚了”；还有人想的是节奏更快、路径更短、不要挡住玩家继续操作。

所以我们以为在讨论方案，其实很多时候只是在交换形容词。大家对上的只是词，还不是画面。

这个现象我想叫它“形容词共振”。听起来大家在同频，但其实每个人脑子里的画面可能差了一个美术版本，甚至差了一套需求文档。

这就是今天想聊 Image 2.0 的原因。

今天不是来讲“AI 多厉害”，也不是来讲“以后 UE 不用想了”。如果这个结论成立，那我们今天应该不是分享会，是职场告别会，气氛会比现在沉重很多。

我真正想讲的是：UE 不是缺少思考，而是很多思考还没被看见。

我们在脑子里已经做了很多判断：这个反馈该轻还是重？这个入口该抢还是藏？这个界面应该服务效率，还是服务情绪？但在项目早期，这些判断经常只能藏在线框、文档和口头描述里。

Image 2.0 的价值，就是帮我们提前截一张“脑内截图”。

这张图不一定最终能用，也不应该直接上线。但它能让大家从“我觉得应该更爽”变成“我们指着这张图看，爽到底爽在哪里”。

方向对不对？情绪到没到？层级清不清楚？值不值得继续投入？

这就是今天的主线：Image 2.0 不替 UE 做 UI，它降低的是表达成本，让团队更早围绕同一个画面讨论。

## 二、定位：它不是更会画，是开始懂界面｜05:00-13:00

> 排练锚点：05:00 定义“界面假设生成器”；08:00 讲到“UE 可讨论线”；10:30 P11 内部经验一笔带过；12:00 收到“四件事”。

那怎么理解 Image 2.0？

Image 2.0 大家都或多或少见过了，所以这里我不想从“它能生成什么图”开始讲，而是换一个问题：它对 UE 到底有什么用？

我觉得对 UE 来说，它不是一个单纯的画图工具。它不只会画图，而是开始懂界面了，更像是一个“界面假设生成器”。

比如一个很小的需求：“保存配方成功，需要一个反馈。”

这句话看起来很简单，但 UE 脑子里会自动弹出一串问题：这是高频操作还是关键时刻？玩家保存完是不是还要继续编辑？这是普通保存，还是第一次解锁稀有配方？这个反馈应该轻轻提示一下，还是应该制造一个小小的完成感？

过去我们要么口头解释，要么手拼参考，要么去翻竞品截图。现在 Image 2.0 可以先帮我们把几种假设画出来：轻量 Toast、局部发光、重度成功面板，甚至带一点仪式感的演出包装。

注意，它生成的不是答案图，而是“假设图”。

答案图意味着它说了算；假设图意味着我们来审。

这里也可以顺便说一下它和其他工具的差异。

Stable Diffusion 更像可控画室，适合精细控制模型、参数和流程。Midjourney 很像灵感放大器，适合做氛围、概念和情绪表达。Nano Banana 更像快速改图器，适合做变体、融合参考和局部调整。

Image 2.0 对 UE 特别有意思的地方，是它更靠近界面讨论现场。

它开始能处理活动页、抽卡页、充值页、奖励结算、成功反馈这些游戏 UI 场景。更重要的是，它不只是给一张漂亮图，它能保留导航、资源栏、CTA、奖励层级、功能区这些我们真正关心的结构。

所以我觉得它跨过了一条线，叫“UE 可讨论线”。

这条线下面是什么？线框图有结构，但太抽象，很多情绪、资源感、视觉语气讨论不起来。概念氛围图很好看，但它经常和真实界面结构离得太远，团队只能说“感觉对”，但不知道怎么落。

跨过这条线之后，图开始能被指着讨论。

比如抽卡界面，不只是角色好看，还能看卡池侧栏、货币区、十连按钮、奖励层级能不能成立。充值弹窗也不只是活动氛围强不强，还能看价格锚点、礼包卡片、主 CTA 是不是清楚。

对 UE 来说，这就很关键。

因为我们不是为了得到一张“最好看的图”，而是为了尽早得到一张“能暴露问题的图”。

内部经验里也能看到这个变化：它已经不只是生成一张“好看的参考图”了。

海报排版、角色风格转 VI、线框转活动页、批量产图和动效帧，这些产物开始能拿来讨论版式、标题字、信息层级、按钮语气、奖励框、toast、弹窗和关键帧。

这里不用展开太多，先记住一个判断就好：Image 2.0 的界面感更稳定了，也更适合批量发散。但它的定位仍然是沟通参考，细节、本地化、局部改图和专业落地，还是要人工接住。

所以过了可讨论线以后，我们不是只有一张图可以聊，而是开始有一组材料可以帮助团队对齐。

所以 Image 2.0 对 UE 的价值，可以收成四件事。

第一，方向探索。比如轻反馈还是重包装，先跑几条路，看哪条值得继续。

第二，参考拆解。不是说“像这个”，而是拆出布局、层级、节奏、组件语言里哪些东西可以迁移。

第三，分镜沟通。很多反馈不是一张静态图能讲清楚的，需要触发、出现、确认、回流这些关键帧。

第四，表现补足。线框能讲结构，但讲不出成功感、奖励感、解锁感、仪式感，Image 2.0 可以先补一版让团队看见。

所以会用 Image 2.0 的 UE，不是更会画图，而是更会把问题变成画面。

看保存配方这个小例子就很直观。

同样是“保存成功”，如果它是高频操作，玩家还要继续编辑，那我们就应该轻一点。飘字、Toast、轻微发光，短暂停留后自动消失。核心判断是：确认即可，不要打断。

但如果它是首次保存重要配方，或者解锁稀有配方，那它就不只是保存成功了，它是一个小成就。书本飞入、翻页记录、成功面板，这时候打断是有意义的。核心判断变成：把保存包装成一个值得被记住的时刻。

这就是 Image 2.0 带来的变化。

以前我们说“反馈强一点”，大家继续脑补。现在我们把轻和重都画出来，然后讨论：这件事到底值不值得打断玩家？

再看参考拆解这个例子。

“参考给质感，Figma 给结构，Image 2.0 先把感受对齐。”这句话很重要。

我们不是把竞品图拿来照搬，而是先抓它的按钮语气、反馈强度、视觉重心，再把这些质感放回我们自己的结构里。

这样讨论就从“像不像某个游戏”变成“这种质感放进我们的信息层级里，到底合不合适”。

这一步其实很 UE。因为我们不是在追一张漂亮皮肤，而是在验证一种表现语气能不能服务自己的结构。

## 三、方法论：不要念咒，要发 Brief｜13:00-22:00

> 排练锚点：13:00 Brief 方法；15:00 过去手拼成本；16:30 循环签到；18:00 线框 / Prompt / 参考图三法；20:00 好友赠礼。

接下来讲怎么用。

第一个误区：不要把 AI 当许愿池。

“史诗感、8K、大师作品、高级、氛围拉满”，这些词不是完全不能用，但如果只给这些，AI 很容易回你一张视觉很热闹、但交互上完全用不了的图。

对 UE 来说，真正有价值的输入不是咒语，而是一份设计 Brief。

这份 Brief 至少要讲清楚几件事。

我要生成什么界面？玩家此刻要完成什么动作？项目是什么类型？世界观和用户语气是什么？输入材料有哪些？线框、竞品、现有 UI、文案草稿、关键数据分别扮演什么角色？页面结构里哪些模块必须保留？主操作区在哪里？信息区在哪里？状态反馈区在哪里？输出时要避免什么？

Image 2.0 是一个好的执行者。Brief 越清楚、越完整，它越少乱跑，迭代次数也会少很多。

我很喜欢这句：线框锁住能用，Prompt 定义好不好。

线框告诉 AI：别乱摆。Prompt 告诉 AI：别乱想。

我们回头看以前的成本。很多时候我们不是不会判断，而是为了把判断说清楚，得先手拼一张图。

找素材、抠图、调色、缝合、补光效、临时改层级。做完以后还要加一句：“大家注意，这个不是最终效果，只是示意一下感觉。”

这句话大家应该都说过。说的时候很心虚，因为我们知道对方可能还是会被素材带跑。

所以 Image 2.0 不是帮我们省掉判断，而是把那些低价值的手工拼贴成本往前推，让我们把精力放回判断上。

循环签到这个案例很典型。

UE 图能证明结构可用：签到奖励、循环状态、入口关系都能讲清楚。黑白稿不用画得很满，只要把结构逻辑讲清楚。Image 2.0 把活动感画出来之后，讨论发生了变化。

大家不再只说“逻辑能跑”，而是开始看：活动感够不够？奖励是不是抢？视觉是不是过重？这个氛围是不是会影响日常签到的轻量感？

也就是说，风险被提前了。

这才是生成图最有价值的地方：不是让我们晚点发现问题，而是早点暴露问题。

接着看线框、Prompt 和参考图的三套方法。

第一，线框也是输入。线框不是低级材料，它是 UE 的秩序。它锁住入口、难度、奖励、层级和交互约束。参考图负责提供材质、光影、UI 语气。Image 2.0 负责把它们合成一个可讨论的世界。

第二，Prompt 是给线框补语境。比如“配方已满不是错误弹窗，而是一次厚重的取舍”。这句就不是普通风格词，它是灵魂指令。它告诉 AI，这不是红色错误提示，不是冷冰冰的系统阻断，而是一个需要让玩家认真选择的时刻。

第三，参考图要先拆层。参考布局层、组件层、视觉层，也要明确风险层。具体角色、Logo、文案、过度相似的构图不能碰。我们取逻辑，不取皮肤。

这也是为什么“看图不是选美，是分配精力”。

我们不应该问：哪张最好看？

应该问：哪张值得投入？

轻度需求，比如保存反馈，高频、工具型，UE 精力可能只需要 5%。快速确认，不阻塞流程。

中度需求，比如模式选择，核心、认知型，UE 精力可能是 30%。这里要严审信息层级，守住交互红线。

重度需求，比如拆信、赠礼、抽卡爆点、首次解锁，峰值、情感型，UE 精力可能要 65%。这时候要评估资源成本，推演动效还原度，因为它真的会影响玩家记忆点。

这不是懒，这是设计精力管理。

我们不是每个地方都卷到满格。高频操作要顺，情感峰值要重，资源要花在能产生体验价值的地方。

好友赠礼案例就是这个方法的落地。

一开始只有 5 分钟线框，它能回答结构问题：按钮在哪、礼物在哪、确认流程怎么走。但它回答不了仪式感。

于是我们把线框作为骨架，把风格参考作为语气，用 Image 2.0 先定视觉基调，再回 Figma 调整细节。然后用 Nano Banana 做局部关键帧修正，再通过视频生成验证动效节奏。

这里最重要的判断是：同样是赠礼，送礼和收礼不是同一种体验。

送礼反馈的关键词是社交效率。它要快，不阻塞，支持连续操作。玩家送完一个，还可能继续送下一个，所以不能每次都给他办一场典礼。

收礼拆信的关键词是情感峰值。它可以重一点，因为这是“我被别人惦记了”的时刻。信封、光影、礼物揭示、确认收礼，这些东西服务的是期待和惊喜。

如果只写文档，我们可能要写很多句：“送礼轻量，收礼重度；送礼偏效率，收礼偏情绪。”但画面和关键帧出来以后，团队会更快懂。

这就是我觉得很关键的一点：能用画面解决的共识，不要全靠长文字硬扛。

## 四、重度案例：烹饪系统里的体验导演｜22:00-29:00

> 排练锚点：22:00 烹饪系统复杂度；23:00 主界面发散；24:00 换锅具资源清单；25:00 表演分镜；26:00 图鉴；27:00 流水席；28:00 大厅入口和跨职能落地。

如果说好友赠礼是一个单点体验，那烹饪系统就是一个长链路系统。

它不是一个界面，而是一组强表现体验。

从局内食材获取，到局外烹饪台；从食材选择、锅具推荐、烹饪演出，到便当承接、战备携带，最后还要在出生岛吃饭让 Buff 生效。

这个系统牵涉的职能很多：UI、原画、模型、动作、动效、特效，每个人都需要理解同一个体验目标。

如果只靠策划文档和 Figma 结构稿，结构能讲清楚，但表现很难讲清楚。火候、蒸汽、锅具材质、菜品揭示、Buff 飞入、角色吃饭镜头，这些很难靠“更沉浸一点”解决。

所以这里 Image 2.0 的第一步，是替 Figma 探路。

先发散布局。沉浸式全屏、侧边栏式、面板式。AI 先把路口照亮，UE 再决定哪条路值得继续走。

这里不要误会，不是 AI 决定方案。AI 只是把路口照亮一点，真正决定往哪走的还是 UE。

接着看换锅具评估。

换锅具这个环节很有意思，因为它把“好看”变成了资源清单。

锅具设定、背景发散、定稿与锅具包、动效验证。到了这一步，我们讨论的就不只是“这个锅好不好看”，而是它背后的资源问题：模型规格是什么？切换节奏怎么做？特效类型有哪些？资源开销能不能承受？

资源问题越早看见，方案越早可控。

这也是 Image 2.0 对复杂系统很实际的价值。它不是让我们在早期把所有图都画漂亮，而是把隐性的资源成本提前逼到桌面上。

表演分镜也是同理。

准备、加热、爆点、结算承接。四个关键帧一摆出来，原画、特效、动效和 UI 就知道该看什么：原画看构图和氛围，特效看火焰、烟雾、光效强度，动效看节奏，UI 看结果面板怎么承接。

看见关键帧，才能判断值不值得投入。

再往下看图鉴内容资产。

烹饪系统不是只有“做饭”这一个动作，它还会沉淀很多内容：食材、配方、世界观说明、图鉴漫画、解锁记录。这些内容如果只是丢在策划表里，玩家看到的是信息；但如果 UE 把它重新组织成图鉴体验，玩家看到的就是世界观。

所以这里我很喜欢这句话：策划铺内容，UE 搭舞台。

左边是策划输出的批量内容资产，食材、世界观、图鉴漫画都在里面。Image 2.0 可以帮助我们把这些内容先包装成更有叙事感的图鉴材料。右边是 UE 要做的转化：它不是把内容平铺出来，而是要设计交互框架、解锁演出和视频预演。

图鉴看起来像信息入口，但它其实也是世界观入口。玩家不是在查表，他是在一点点打开这个烹饪系统背后的文化、食材和故事。这个时候 Image 2.0 帮忙补的不是装饰，而是“这个内容怎么被玩家感知”的第一版舞台。

再看流水席局内状态。

流水席如果只写成功能需求，大概就是三个词：入口确认、吃饭进度、Buff 生效。听起来非常清楚，但也非常干。问题是玩家真正感受到的，不是这三个功能名，而是吃饭前、吃饭中、吃饭后的连续情境。

所以这里先用参考图和 Image 2.0 把三段场景定出来：吃饭前，玩家要确认入口，知道自己即将进入一个可交互的用餐状态；吃饭中，玩家要看到进度和状态，知道自己正在做什么、还要多久、有没有被打断；吃饭后，玩家要明确看到 Buff 生效，知道这顿饭和后续战斗准备发生了关系。

这一步的价值，是把“功能状态”放回“场景情境”里。先有分镜，再有 UI 状态。否则我们很容易只做出三个按钮和三个提示，但玩家感受不到这件事为什么属于烹饪系统。

所以这里的逻辑是：分镜定情境，UE 把情境翻译成功能状态。

再往后，大厅入口和动作资源也很关键。

入口不是一个按钮。至少在这个系统里，它不应该只是一个按钮，而是一段场景动作。

如果烹饪系统要成立，玩家从大厅进入烹饪台的过程，本身就是体验的一部分。空间关系在哪里？火光在哪里？桌面和锅具怎么摆？角色站在哪里？手部怎么接触锅具？烹饪姿态是什么？这些东西都不是“上线前再补一下”的装饰，它们会直接影响模型、动作、镜头和资源评估。

所以把大厅场景资源、烹饪动作资源和动作视频放在一起看，团队就能很快意识到：这时候讨论的就不是“按钮放哪”，而是这段体验做不做得起。

这也是烹饪系统作为重点案例最有代表性的地方。Image 2.0 不只是帮 UE 画出界面方案，它还帮 UE 把一个系统里那些容易被忽略的表现节点提前摆出来：内容怎么变成图鉴，功能怎么变成局内状态，入口怎么变成场景动作。

AI 先帮我们定情境，UE 再把情境翻译成功能、资源和执行判断。

到了跨职能落地这个阶段，UE 的角色就很清楚了。

同一张图，不同职能看的不是同一件事。UE 要做的，就是把它翻译成各职能能执行、能评估的判断。

UI 看结构和状态：入口、面板、Buff 显示、结果承接和按钮反馈。

原画看构图和色调：大厅空间、火光、桌面关系、世界观氛围。

模型和动作看资源点：角色站位、手部交互、桌椅锅具、进食动作。

动效和特效看节奏：烟火浓度、Buff 飞入、转场衔接、结果爆点。

所以在烹饪系统里，Image 2.0 的价值不只是生成 UI 图，而是把策划规则和 Figma 交互稿之间缺少的表现画面补出来。

这个时候 UE 不只是“画界面的人”，更像一个体验导演和跨职能翻译官。

这个说法听起来有点大，但其实很朴素：我们要把体验目标翻译成每个职能听得懂、做得下去、评估得了的材料。

## 五、扩展：当它不只是一张图｜29:00-32:00

> 排练锚点：29:00 从零定义世界；30:00 视觉自助餐；31:00 组合工作流和 X 案例快速带过。

再往外看，当 Image 2.0 不只是一张图，会发生什么？

比如从零定义一个世界。

过去我们经常把大量竞品 UI 参考、元素拼贴、核心概念提案交给下游，让大家吸收。不是没有努力，而是输入太散，团队很难同时脑补到同一张图。

所以这里的 game-ui-concept-board 案例很有启发。

它把主观的“感觉”拆成 5 步推演：世界观和 IP 理解、字体方案、配色方案、设计语言、综合方案。

这件事的价值不是“AI 又生成了一堆图”，而是把脑洞工程化了。

同一个 IP，三套方向不能只是三套配色。方向差异应该来自字体结构、色彩比例、组件语言、构图方式和叙事语气。

比如海贼冒险世界，可以有伟大航路海图方向，强调探索、航线、地图和指南针；也可以有悬赏漫画爆发方向，强调节奏、冲击字和追击反馈；还可以有阳光船团宴会方向，强调社交、伙伴、宴会和轻松叙事。

这就像给团队一张“视觉自助餐”。

不是说大家直接端一盘上线，而是三套方向摆在桌上，方便挑选、混搭、深化。原本说不清的感觉，变成了能拆开看的画面。

接着进入组合工作流。

静态图定方向，但有些体验只看单张图还不够，所以后面要让画面动起来，继续验证节奏。

Image 2.0 负责界面状态和视觉方向。Nano Banana 可以做风格转化和局部修改。Seedance 让状态动起来。LibTV 负责把素材、脚本、分镜和生成节点组织起来。

这里有个非常重要的边界：这不是生成可玩的游戏。

它生成的是一段可以讨论、复盘和复用的体验证据。

X 案例里，流程从 2D 角色设定开始，再转成 3D 质感，然后用 Image 2.0 生成 starting、customization、loading、in-game 等界面状态，最后把这些状态拆成时间轴脚本，交给 Seedance 做连续镜头。

对 UE 的启发是：一张图不是终点，它可以继续变成界面状态、脚本和视频片段。Image 2.0 不孤立出图，它可以进入一条可讨论流程。

它负责把界面状态画清楚，Seedance 负责让状态之间的变化动起来，LibTV 负责把过程组织起来。

看见方向，也看见过程。

## 六、结尾：旧经验不会失效，只是判断对象变了｜32:00-34:00

> 排练锚点：32:00 UE 基本盘；33:00 方向盘没有丢；33:30 收到“让思考被看见”。

最后我们要把工具收回来，回到 UE 自己。

AI 能跑出很多可能性，但交互设计师的基本盘不会失效。

信息架构谁来判断？谁先出现，谁服务谁，哪些信息必须让玩家第一眼读到，这不是一张漂亮图能替我们决定的。

状态反馈谁来判断？成功、失败、等待、打断、恢复，每个状态要怎么接，仍然要 UE 设计。

路径设计谁来判断？入口、回退、确认、取消、下一步，不能只看单张图是否好看。

异常兜底谁来判断？弱网、空状态、资源不足、超时、多端适配，这些都是项目里真正会咬人的问题。

所以不用慌，方向盘没有丢。

只是我们手里多了一个更快的方式，把判断对象提前画出来。

以前很多 UE 的思考藏在脑子里、藏在线框里、藏在文档里。要等 UI、美术、动效投入之后，团队才慢慢发现：哦，原来我们想的不一样。

现在我们可以更早把方向画出来，让团队一起看见它、讨论它、修正它。

这就是我今天最想留下的一句话：

Image 2.0 的价值，不是让 UE 少思考，而是让思考被看见。

不是少思考，是让判断更早发生。

谢谢大家。
